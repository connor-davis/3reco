# Internal Actions

Some operations in 3rEco involve side effects that cannot run inside a database transaction — like generating a PDF or calling an external service. These use Convex **internal actions**, which can call external APIs and access Convex storage.

## Function Visibility

Convex has two visibility levels:

| Prefix | Callable from | Used for |
|---|---|---|
| `query` / `mutation` / `action` | Frontend clients | Public API |
| `internalQuery` / `internalMutation` / `internalAction` | Other Convex functions only | Server-side helpers, side effects |

Internal functions are never exposed to the frontend. They are invoked with `ctx.runQuery`, `ctx.runMutation`, or `ctx.scheduler.runAfter` from other Convex functions.

## The Invoice Generation Flow

PDF invoices are generated by an internal action in `convex/invoices.ts`. The flow illustrates the typical internal-action pattern:

```
1. acceptOffer mutation completes the transaction
        ↓
2. Transaction document is inserted into DB
        ↓
3. Mutation schedules the internal action:
   ctx.scheduler.runAfter(0, internal.invoices.generateForTransaction, { transactionId })
        ↓
4. Internal action runs asynchronously:
   a. internalQuery reads transaction, seller, buyer, material
   b. pdf-lib generates the PDF in memory
   c. ctx.storage.store(blob) saves it to Convex file storage
   d. internalMutation patches transaction.invoiceStorageId with the storage ID
        ↓
5. Frontend queries api.invoices.getInvoiceUrl — returns a signed download URL
```

### Why `runAfter(0, ...)` instead of `await` inside the mutation?

Actions cannot run inside a transaction. `ctx.scheduler.runAfter(0, ...)` schedules the action to run immediately after the current mutation commits, keeping the mutation transactional while still triggering the side effect.

## Writing an Internal Action

```typescript
import { internalAction, internalMutation, internalQuery } from './_generated/server'
import { internal } from './_generated/api'
import { v } from 'convex/values'

// Step 1 — helper query to read data
export const getDataForReport = internalQuery({
  args: { id: v.id('transactions') },
  handler: async (ctx, { id }) => {
    return await ctx.db.get('transactions', id)
  },
})

// Step 2 — helper mutation to write the result back
export const patchReportId = internalMutation({
  args: { id: v.id('transactions'), reportId: v.string() },
  handler: async (ctx, { id, reportId }) => {
    await ctx.db.patch('transactions', id, { /* ... */ })
  },
})

// Step 3 — the action that orchestrates the flow
export const generateReport = internalAction({
  args: { id: v.id('transactions') },
  handler: async (ctx, { id }) => {
    // Read data via internal query
    const data = await ctx.runQuery(internal.reports.getDataForReport, { id })
    if (!data) return

    // Do async work (external API, PDF generation, etc.)
    const result = await someAsyncWork(data)

    // Write result back via internal mutation
    await ctx.runMutation(internal.reports.patchReportId, {
      id,
      reportId: result.id,
    })
  },
})
```

## Convex Storage

Use `ctx.storage` inside actions to store and retrieve files:

```typescript
// Store a file
const blob = new Blob([bytes], { type: 'application/pdf' })
const storageId = await ctx.storage.store(blob)  // returns Id<'_storage'>

// Get a signed download URL (in a query)
const url = await ctx.storage.getUrl(storageId)   // expires after ~1 hour
```

Store the `storageId` on a document so you can generate fresh URLs on demand — the URL itself expires but the stored file does not.

## Scheduling vs Direct Call

| Method | When to use |
|---|---|
| `ctx.scheduler.runAfter(0, fn, args)` | Trigger from a mutation — runs after commit |
| `ctx.scheduler.runAt(timestamp, fn, args)` | Delayed execution |
| `await ctx.runQuery(fn, args)` | Inside an action — synchronous read |
| `await ctx.runMutation(fn, args)` | Inside an action — synchronous write |

<Callout type="warning" title="Actions are not transactional">
  If an action fails halfway through (e.g. PDF generation succeeds but the patch mutation fails), you may end up with orphaned storage objects. Design actions to be idempotent where possible, or use retry logic.
</Callout>
